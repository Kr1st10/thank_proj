# Техническое задание

## 1. Назначение системы

Система совместной работы предназначена для хранения, обсуждения и приоритизации предложений по улучшению процессов разработки ПО внутри команды.

Продукт должен помогать команде:
- быстро фиксировать идеи и проблемы процессов;
- обсуждать предложения асинхронно;
- голосовать за важные изменения;
- отслеживать статус внедрения предложений;
- работать с несколькими проектами и составом участников;
- не терять неотправленные данные за счет черновиков, синхронизируемых между устройствами.

## 2. Цели первого дедлайна

На первом дедлайне подготавливается backend-пакет проектирования, который фиксирует:
- функциональные границы расширенного MVP;
- структуру доменной модели;
- REST API v1;
- состав основных сущностей и связей в БД;
- базовую схему авторизации и токенной сессии.

Первый дедлайн не предполагает runnable backend и не требует реализации на C#.

## 3. Роли

### 3.1 Пользователь проекта

Пользователь может:
- войти в систему;
- видеть список доступных проектов;
- выбирать текущий проект;
- просматривать предложения по вкладкам статусов;
- создавать и редактировать собственные предложения;
- просматривать карточку предложения;
- голосовать за или против предложения;
- отменять или менять свой голос;
- писать, редактировать и удалять свои комментарии;
- сохранять черновики предложений и комментариев.

### 3.2 Администратор проекта

Администратор проекта обладает всеми правами обычного пользователя и дополнительно может:
- добавлять участников в проект;
- удалять участников из проекта;
- назначать роль администратора в рамках проекта;
- просматривать и управлять составом проекта.

В модели первого дедлайна администраторская роль задается на уровне участия пользователя в проекте.

## 4. Основные сущности

- `Проект` — рабочее пространство, в рамках которого ведутся предложения.
- `Участник проекта` — связь пользователя с проектом и его ролью.
- `Предложение` — идея или замечание по улучшению процесса.
- `Голос` — голос пользователя за или против предложения.
- `Комментарий` — сообщение в обсуждении предложения, поддерживающее вложенные ответы.
- `Черновик` — неотправленные изменения предложения или комментария.
- `Сессия авторизации` — серверно управляемая refresh-сессия пользователя.
- `Пользователь` — участник системы, автор предложений, комментариев и голосов.

## 5. Основные пользовательские сценарии

### 5.1 Вход в систему

Пользователь проходит аутентификацию и получает `access token` и `refresh token`. Дальнейшая работа с защищенными endpoint-ами выполняется по Bearer Token.

### 5.2 Выбор проекта

Пользователь открывает приложение, получает список доступных проектов и выбирает проект, в рамках которого хочет работать.

### 5.3 Просмотр списка предложений

Пользователь открывает проект и видит список предложений, разбитый по статусам:
- `New`
- `InProgress`
- `Accepted`
- `Rejected`

Для каждого предложения отображаются:
- текст предложения;
- автор;
- суммарный результат голосования;
- дата и время создания.

### 5.4 Создание предложения

Пользователь нажимает «Добавить предложение», вводит текст и отправляет форму. Предложение сохраняется в системе со статусом `New`.

### 5.5 Просмотр карточки предложения

Пользователь открывает карточку предложения и видит:
- текст предложения;
- автора;
- статус;
- агрегированный итог голосов;
- свой текущий голос;
- детализацию по голосам;
- комментарии;
- связанные черновики текущего пользователя.

### 5.6 Голосование

Пользователь может:
- проголосовать `за`;
- проголосовать `против`;
- поменять тип голоса;
- отменить голос.

### 5.7 Обсуждение

Пользователь может:
- добавить комментарий к предложению;
- ответить на существующий комментарий;
- отредактировать свой комментарий;
- удалить свой комментарий.

Комментарии отображаются в хронологическом порядке. Ответы отображаются в древовидной структуре интерфейса.

На уровне API допускается возврат комментариев плоским списком с `parentCommentId`, а дерево строится на клиенте.

### 5.8 Работа с черновиками

Если пользователь ввел текст предложения или комментария, но не отправил его, система сохраняет введенные данные как черновик.

Черновики должны:
- храниться отдельно от опубликованных сущностей;
- принадлежать конкретному пользователю и проекту;
- синхронизироваться между устройствами пользователя;
- быть доступны для продолжения редактирования.

### 5.9 Управление участниками проекта

Администратор проекта может:
- добавить пользователя в проект;
- удалить пользователя из проекта;
- изменить роль участника проекта.

## 6. Бизнес-правила

### 6.1 Статусы предложений

В расширенном MVP используются статусы:
- `New`
- `InProgress`
- `Accepted`
- `Rejected`

### 6.2 Сортировка списка

Список предложений внутри вкладки статуса сортируется:
1. по суммарному значению голосов по убыванию;
2. при одинаковой сумме — по времени создания по возрастанию, то есть более старое предложение выводится выше.

### 6.3 Правила голосования

- Один пользователь может иметь не более одного активного голоса на одно предложение.
- Допустимые типы голоса: `up` и `down`.
- Пользователь может заменить один тип голоса другим.
- Пользователь может удалить свой голос.
- Агрегированная сумма голосов является вычисляемой величиной.
- `score = количество голосов Up - количество голосов Down`.

### 6.4 Правила комментариев

- Комментарий всегда принадлежит конкретному предложению.
- Комментарий может быть корневым или ответом на другой комментарий.
- Только автор комментария может редактировать и удалять его.
- Удаление комментария не должно нарушать структуру обсуждения. На этапе реализации допустим soft delete.
- Для MVP backend возвращает комментарии полным плоским списком по предложению, без пагинации.
- Дерево комментариев собирается на клиенте по `parentCommentId`.

### 6.5 Правила черновиков

- Черновик предложения не является опубликованным предложением.
- Черновик комментария не является опубликованным комментарием.
- У пользователя может быть несколько черновиков в рамках проекта.
- Для черновика комментария должна сохраняться связь с предложением и, при необходимости, с родительским комментарием.
- Для MVP допустимо хранить содержимое черновика в JSON-поле, чтобы не усложнять первую реализацию.

### 6.6 Правила доступа

- Пользователь видит только проекты, в которых состоит.
- Пользователь может работать только с сущностями в рамках доступного ему проекта.
- Управление составом проекта доступно только администраторам проекта.

### 6.7 Правила токенной сессии

- После входа пользователь получает `access token` и `refresh token`.
- `access token` используется для доступа к API.
- `refresh token` используется для продления access token.
- Logout должен отзывать refresh-сессию на стороне backend.
- Refresh token не должен храниться в открытом виде.
- Для MVP у пользователя допускается только одна активная refresh-сессия.
- Новый login инвалидирует предыдущую refresh-сессию.
- При refresh происходит ротация refresh token.
- Рекомендуемые сроки жизни: `access token = 15 минут`, `refresh token = 7 дней`.

### 6.8 Правила редактирования предложений

- Автор предложения может редактировать только текст предложения.
- Администратор проекта может изменять только статус предложения.
- Для MVP эти действия разделяются на уровне API, чтобы упростить авторизацию и backend-логику.

## 7. Авторизация

### 7.1 Целевой вариант

В целевом продукте пользователи авторизуются под доменной учетной записью, а список пользователей для добавления в проекты подтягивается из доменной директории.

### 7.2 Подход для учебного MVP

Для учебного MVP принимается схема `adapter + dev login`.

Это означает:
- бизнес-логика не зависит напрямую от конкретной корпоративной системы входа;
- backend проектируется через абстракцию получения текущего пользователя;
- в учебной реализации допустим login по email/password через dev auth-provider;
- реальная доменная интеграция может быть подключена позже без переделки доменных сценариев и REST API.

## 8. Нефункциональные требования

- Документы и контракты должны храниться в git и быть удобны для ревью.
- API должен быть пригоден для последующего описания в OpenAPI/Swagger.
- Модель должна быть расширяемой под реальные корпоративные интеграции.
- Все даты и время должны храниться в UTC.
- Идентификаторы сущностей предполагаются UUID.
- API версионируется через префикс `/api/v1`.

## 9. Границы первого дедлайна

В первый дедлайн входят:
- login, refresh и logout как часть API-контракта;
- список проектов;
- создание проекта;
- управление участниками проекта;
- список предложений;
- создание и редактирование предложений;
- просмотр карточки предложения;
- голосование;
- комментарии;
- черновики предложений и комментариев;
- проектирование БД и API.

В первый дедлайн не входят как обязательная реализация:
- настройка количества голосов на пользователя;
- периодический сброс лимитов голосов;
- расписание встреч;
- интеграция с Jira;
- интеграция с Outlook;
- реальная интеграция с доменной директорией;
- runnable backend;
- инфраструктурный код и миграции.

## 10. Критерии готовности документации

Документация первого дедлайна считается готовой, если:
- ТЗ описывает роли, сценарии и бизнес-правила без противоречий;
- ER-диаграмма покрывает сущности расширенного MVP, включая auth-session;
- REST API согласован с ТЗ и ER-моделью;
- future scope явно отделен от текущего объема первого дедлайна.

## 11. User Stories

- Как пользователь проекта, я хочу войти в систему, чтобы получить доступ к своим проектам и предложениям команды.
- Как пользователь проекта, я хочу видеть список доступных мне проектов, чтобы быстро переключаться между рабочими пространствами.
- Как пользователь проекта, я хочу видеть список предложений по статусам, чтобы понимать, какие идеи новые, какие уже внедряются, а какие завершены.
- Как пользователь проекта, я хочу создавать новое предложение, чтобы зафиксировать идею по улучшению процесса сразу, когда она появилась.
- Как автор предложения, я хочу редактировать текст своего предложения, чтобы уточнять формулировку после обсуждения.
- Как пользователь проекта, я хочу голосовать за или против предложения, чтобы показывать его важность для команды.
- Как пользователь проекта, я хочу отменять или менять свой голос, чтобы корректировать свою позицию после обсуждения.
- Как пользователь проекта, я хочу открыть карточку предложения, чтобы увидеть его статус, текущий результат голосования и детали обсуждения.
- Как пользователь проекта, я хочу добавлять комментарии и ответы, чтобы обсуждать предложение с коллегами в контексте.
- Как автор комментария, я хочу редактировать или удалять свой комментарий, чтобы исправлять ошибки и поддерживать обсуждение в актуальном состоянии.
- Как пользователь проекта, я хочу сохранять черновики предложений и комментариев, чтобы не потерять неотправленный текст.
- Как администратор проекта, я хочу добавлять и удалять участников проекта, чтобы поддерживать актуальный состав команды.
- Как администратор проекта, я хочу менять роль участника проекта, чтобы делегировать управление другим членам команды.
- Как администратор проекта, я хочу менять статус предложения, чтобы отражать текущее состояние его внедрения.
